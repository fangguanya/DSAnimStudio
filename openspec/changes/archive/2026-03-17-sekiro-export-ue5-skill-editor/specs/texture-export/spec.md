## 新增需求

### 需求:TPF纹理到标准格式导出
系统必须能从 TPF (Texture Package Format) 归档文件中提取纹理，并导出为标准图像格式。

#### 场景:导出为DDS格式
- **当** 用户选择 DDS 导出格式
- **那么** 系统必须直接写出 `TPF.Texture.Bytes`（原始 DDS 数据），保持原始压缩格式不变，文件扩展名为 `.dds`

#### 场景:导出为PNG格式
- **当** 用户选择 PNG 导出格式
- **那么** 系统必须使用 Pfim 库解码 BC 压缩纹理（DXT1/DXT3/DXT5/BC4/BC5/BC7）为 RGBA 像素数据，然后编码为 PNG 文件

#### 场景:处理不支持的纹理格式
- **当** 纹理使用 Pfim 不支持的格式（如 BC6H）
- **那么** 系统必须跳过该纹理，记录警告日志包含纹理名称和格式类型，并继续处理剩余纹理

### 需求:从游戏归档提取纹理
系统必须支持从 Sekiro 游戏目录的各类归档文件中定位和提取角色纹理。

#### 场景:从CHRBND提取内嵌纹理
- **当** 角色的 `.chrbnd.dcx` 文件内包含 TPF 纹理
- **那么** 系统必须解包 BND4 归档，识别其中的 TPF 文件并提取所有纹理

#### 场景:从独立TPF文件提取纹理
- **当** 纹理以独立的 `.tpf.dcx` 文件存在于游戏目录中
- **那么** 系统必须正确解压 DCX 压缩，解析 TPF 格式并提取所有纹理

### 需求:纹理命名规范
导出的纹理文件必须使用可追溯的命名规范。

#### 场景:保留原始纹理名称
- **当** 导出纹理文件
- **那么** 文件名必须使用 TPF.Texture 中的原始纹理名称（如 `c0000_body_a`），加上目标格式扩展名（`.png` 或 `.dds`）

#### 场景:处理重名纹理
- **当** 同一角色存在重名纹理（来自不同 TPF 包）
- **那么** 系统必须在文件名后追加数字后缀（如 `_01`、`_02`）以避免覆盖
