## ADDED Requirements

### 需求:正式导出物必须使用唯一交付格式
Sekiro 正式导出链路对角色模型与动画必须仅输出 `glTF 2.0 (.gltf + .bin)`，对纹理必须仅输出 `PNG`。正式交付物禁止包含 `fbx`、`fbxa`、`glb`、`dae/collada` 或 `dds` 作为成功结果。

#### 场景:角色正式导出成功
- **当** CLI 以正式导出模式导出一个角色
- **那么** 模型目录必须只包含正式模型 `gltf/bin` 交付物，动画目录必须只包含正式动画 `gltf/bin` 交付物，纹理目录必须只包含 `png` 交付物

### 需求:正式导出链路必须 fail-closed
正式导出链路在任一正式步骤失败时必须显式失败并记录原因，禁止通过回退格式、占位数据、静默跳过或 warning-only 继续将该资产记为成功。

#### 场景:任一步骤失败
- **当** HKX 解析、骨骼构建、动画采样、glTF 结构化输出、纹理解码、模板读取或参数读取失败
- **那么** 当前资产必须被标记为失败，并进入角色级验收报告

### 需求:正式导出必须区分规范化装配与编辑器预览装配
正式导出合同必须把最终角色模型定义为规范化装配结果，而不是 DSAnimStudio 当前 UI 会话中的临时预览状态。正式链路不得把 `ChrAsm` 的任意交互式状态直接视为正式交付定义。

#### 场景:导出主角等多部件角色
- **当** 正式链路导出需要通过多部件装配才能形成最终角色模型
- **那么** 导出器必须使用已声明的 formal assembly profile，并在验收报告中可追溯该装配来源

### 需求:导出修补步骤不得掩盖无效交付物
如果正式链路仍包含导出后归一化或修补步骤，该步骤必须被视为正式流水线的一部分并且失败即中止；禁止产生未完成修补的交付物。

#### 场景:归一化步骤失败
- **当** glTF 输出需要进行 buffer、skin、animation 或 scene 归一化且该步骤失败
- **那么** 正式导出必须失败，且不得留下被记为成功的 glTF 交付物

### 需求:正式输出与兼容输出必须严格隔离
如果工具仍保留调试或兼容导出模式，这些输出必须与正式交付目录、正式统计和正式验收严格隔离。正式模式不得因为兼容输出存在而被记为成功。

#### 场景:同时存在正式模式和兼容模式
- **当** CLI 支持额外的 debug/compat 导出选项
- **那么** 这些产物不得写入正式目录，也不得影响正式角色成功统计
