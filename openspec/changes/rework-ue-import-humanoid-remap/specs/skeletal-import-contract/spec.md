## ADDED Requirements

### 需求:UE 导入必须验证归一化后的人形结果
UE 侧正式导入检查必须针对归一化后的 Skeleton、SkeletalMesh 和动画结果执行，而不能只检查源 glTF 数据能否被直接落盘为可见或可绑定资产；如果需要进一步验收，必须在 import 完成后按需单独执行检查，而不是依赖独立冗余 commandlet 或内嵌历史验证器。

#### 场景:验证归一化后的正式导入产物
- **当** UE 导入器完成一个角色的模型、骨架和动画归一化发布
- **那么** 它必须基于归一化后的资产执行骨架层级、横向镜像结果、朝向和动画绑定覆盖检查，并把该结果作为正式成功判定依据

## MODIFIED Requirements

### 需求:UE 导入必须验证 formal skeleton root 与动画绑定
UE 侧正式导入必须确认模型存在唯一 formal skeleton root，导入链路已按正式归一化配置完成人形骨架重组与坐标系变换，动画 clip 与归一化后的 skeleton 可绑定，且不会依赖历史兼容路径、导入期猜测修补、导入后额外 retarget 步骤，或独立冗余验证 commandlet 才能成立。

#### 场景:导入正式模型与单 clip 动画
- **当** UE 同时处理某角色的正式模型和正式动画交付物
- **那么** 它必须先完成导入侧的人形归一化与骨架重组，再形成可绑定的 Skeleton/Animation 关系，否则该角色不得计为正式导入成功

### 需求:不符合正式合同的输入不得计为导入成功
如果输入资产缺失正式骨架关系、缺失正式动画绑定、缺失正式纹理交付，或者导入结果缺失归一化后的人形骨架组织、逐帧动画重算、统一矩阵变换，或依赖 FBX、DDS、Collada、宽松兼容字段、导入后额外 retarget、独立冗余验证 commandlet 才能成立，则该输入不得被计为正式导入成功。

#### 场景:输入依赖历史兼容路径
- **当** 一个角色资产只能通过历史兼容路径而非正式合同导入 UE
- **那么** 导入验证必须失败并在验收报告中说明原因

#### 场景:输入跳过导入侧归一化
- **当** 一个角色资产的正式导入结果仍然只是基于源 glTF 数据直接生成的未归一化资产，尚未完成导入侧规定的人形骨架重组和动画重算
- **那么** 该角色不得被计为正式导入成功，即使未归一化直出资产已经能被 UE 创建为 Skeleton 或 Animation
