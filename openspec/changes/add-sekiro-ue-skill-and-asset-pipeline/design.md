## 上下文

当前仓库已经具备三类基础能力：

- `SekiroExporter` 和 `DSAnimStudioNETCore/Export/` 可以导出 Sekiro 的模型、动画、纹理、材质清单和技能配置
- `ue5-import-scripts` 规格定义了 UE5 Python 批量导入脚本，但重点仍然是“把文件导进来”
- `ue5-skill-editor-plugin` 规格定义了技能浏览、时间轴和 3D 视口，但对材质实例完整性、参数链路复原和 Sekiro 专有技能语义的验收还不够严格

用户要求的是完整复刻而不是局部演示，因此这次设计必须把导出端、导入端和编辑器端收敛成一条正式链路。链路中的每一层都要输出或消费明确契约，不能再依赖缺省推断、宽松导入、后处理修补或人工兜底。

这项设计跨越多个模块：`SekiroExporter`、导出 JSON/清单 schema、UE Python 导入脚本、`SekiroSkillEditor` 的数据资产与编辑器 UI。它同时涉及新的角色级验收标准，因此需要独立设计文档来固定边界。

## 目标 / 非目标

**目标：**

- 定义一条从 Sekiro 游戏资源到 UE 工程 `SekiroSkillEditor` 的正式资产复刻链路
- 要求导出侧提供完整的角色资产包，包括 UE 重建材质、纹理、动画、技能和参数语义所需的正式元数据
- 要求导入侧在 UE 中建立稳定的资源关系，而不仅仅把文件导入 Content Browser
- 要求插件侧能复刻 Sekiro 技能语义，包括事件、参数链路、DummyPoly 定位、义手覆盖和战技样式检查
- 建立角色级 fail-closed 验收合同，任何关键资产或关键语义缺失都不得记为成功

**非目标：**

- 不在本变更中定义新的游戏设计逻辑或对 Sekiro 原始规则做“增强版”编辑能力
- 不要求在本变更中覆盖非 Sekiro 游戏的通用技能编辑器需求
- 不把历史上非正式的 FBX/DDS 调试路径重新纳入正式完成标准
- 不在本提案阶段细化每一个 UE Slate 组件的具体交互实现

## 决策

### 决策: 将正式链路拆分为“导出资产包 -> UE 导入编排 -> 编辑器复刻与验收”三层

导出端、导入端和插件端承担不同职责，必须各自有独立合同：

- 导出端负责产出完整、稳定、可追溯的正式资产包
- 导入端负责把正式资产包转成 UE 资产，并建立资源关系与索引
- 插件端负责消费这些 UE 资产，完成技能语义复刻、可视化和角色级验收

这样可以避免把逻辑塞进单一工具中，也避免 UE 端通过重新解析原始游戏文件来补导出侧的缺口。

考虑过的替代方案：

- 让 `SekiroSkillEditor` 直接读取导出目录中的 JSON/PNG/glTF 并在运行时解释：被否决，因为这会让编辑器承担导入、缓存和验证三种职责，且难以形成稳定 UE 资产引用
- 继续把 Python 脚本视为“辅助脚本”：被否决，因为用户要求的是完整复刻，导入脚本本身必须成为正式链路的一部分

### 决策: 用角色级“正式资产包”作为导出与导入之间的唯一交接单元

每个角色的正式交付必须包含并对齐以下内容：模型、动画、PNG 纹理、材质清单、技能配置、参数关联摘要和角色级导出摘要。所有文件必须通过稳定命名和显式索引互相关联，避免 UE 侧使用模糊匹配推断资源。

这样做的理由是材质、动画和技能不是独立文件集合，而是共享同一角色上下文。将其视为单个资产包，可以把“完整性”定义为角色级属性，而不是零散文件是否存在。

考虑过的替代方案：

- 让 UE 侧分别扫描 `Model/`、`Textures/`、`Skills/` 目录并自行对齐：被否决，因为目录扫描无法表达稳定的跨文件语义约束，且容易回到猜测式匹配

### 决策: 材质和纹理导入必须由正式清单驱动，而不是按文件名约定兜底

`material_manifest.json` 将被收紧为 UE 材质重建合同的一部分。它必须给出足以创建材质实例、绑定纹理参数、区分法线/高光/发光等槽位的稳定键。`texture-export` 则必须保证导出的 PNG 文件名与清单一一对应。

这样做的理由是用户指出目前缺少材质和纹理导入逻辑；如果仍允许 UE 侧按命名猜测槽位，正式链路仍然是不完整的。

考虑过的替代方案：

- 在 UE 脚本中按 `*_n`、`*_s` 等后缀推断纹理用途：被否决，因为这类规则不是正式数据源，且容易被 Sekiro 特定命名破坏

### 决策: 技能复刻以“事件 -> 参数链路 -> DummyPoly/覆盖结果”作为核心数据模型

UE 侧不只需要原始 TAE 事件表，还需要沿着 `BehaviorParam`、`AtkParam`、`SpEffectParam`、`EquipParamWeapon` 以及义手/战技覆盖关系重建技能语义。插件中的时间轴、属性检查器和 3D 视口都必须以这条正式链路为基础数据源。

这样做的理由是“完整复刻”要求用户能在 UE 里验证实际战斗语义，而不是只查看一份事件列表。

考虑过的替代方案：

- 只导入原始事件参数，视口逻辑再临时解释：被否决，因为这会让解释逻辑散落在多个 UI 组件中，也无法形成统一验收标准

### 决策: 验收采用 fail-closed 的角色级验证器

新增 `sekiro-ue-replica-validation` capability，用来定义一个角色只有在以下三类条件同时成立时才算成功：

- 正式资产包完整
- UE 资产导入完整并建立了正确引用
- 插件能复刻技能语义和关键空间结果

这样做的理由是用户要求“完整复刻”，因此任何缺材质、缺纹理、缺技能资产、缺参数索引或视口无法解释关键 Sekiro 事件的结果都必须硬性失败。

考虑过的替代方案：

- 把验证结果作为警告列表：被否决，因为这会把“不完整但能打开”继续混入正式成功路径

## 风险 / 权衡

- [风险] 导出清单和技能 JSON 需要补充更多正式字段，可能导致现有脚本和插件读取逻辑整体失配 -> 缓解措施：通过增量 specs 明确 schema 变更，并把导出摘要作为版本化契约的一部分
- [风险] UE 导入脚本职责扩大后，失败点会从“导入失败”扩展为“资源关系建立失败” -> 缓解措施：要求脚本输出角色级阶段性结果和失败原因，便于与导出摘要对照
- [风险] Sekiro 技能语义复刻需要覆盖 DummyPoly、义手覆盖和战技样式检查，插件实现复杂度显著上升 -> 缓解措施：在任务阶段先固化数据资产与验证器，再实现时间轴和视口消费层
- [风险] 历史调试资源或不完整导出样本将不再被视为成功输入 -> 缓解措施：明确区分正式资产包与调试产物，必要时允许单独保留非正式工具路径，但不能进入正式完成标准

## 迁移计划

1. 先收紧导出合同和清单/schema，确保角色级正式资产包定义稳定
2. 再收紧 UE Python 导入合同，使其按正式资产包建立 SkeletalMesh、纹理、材质实例、技能资产和参数索引
3. 最后收紧 `SekiroSkillEditor` 插件合同和新增角色级验证器，使编辑器以内建验收方式证明完整复刻
4. 旧的“仅导入部分资产”工作流可以继续作为调试路径存在，但必须被标记为非正式，且不参与角色级成功判定

## Open Questions

- `SekiroSkillEditor` 中正式的 UE Python 脚本目录是放在项目 `Content/Python`、插件目录，还是仓库根目录工具集；实现阶段需要按当前工程结构固定
- 角色级导出摘要是否需要单独引入版本字段和校验和，以支持 UE 端快速检查资产包是否完整；提案阶段先要求“可追溯且角色级完整”，实现时再定最终字段
- 对于材质实例所依赖的 Master Material，是否由仓库内提供唯一正式模板，还是由导入脚本在工程内创建；实现阶段需要选择唯一正式路径，禁止双轨并存

## 继承范围

`formalize-sekiro-export-pipeline` 已经把正式 glTF 导出、骨架根、动画解析和 `c0000` 样本链路固定下来，但仍有一批未完成工作天然属于本变更的 UE 闭环范围，需要在归档后由当前 change 继续承担：

- 纹理正式化尚未完成支持矩阵与稳定命名/材质引用契约，因此本变更里的材质实例重建必须直接继承这部分约束
- UE 导入验收尚未把 SkeletalMesh、Skeleton、PhysicsAsset 和动画绑定全部纳入正式通过条件，因此本变更必须把它们并入 `sekiro-ue-replica-validation`
- 形式化回归仍缺少非 `c0000` 角色样本、旧成功基线替换和正式链路清理，因此本变更需要在端到端批量验收阶段补齐
- 原 change 中与 DSAnimStudio UI、UE 编辑器时间轴/视口和攻击判定复刻相关的未完成验证，应该并入本变更的编辑器复刻验收，而不是在归档后丢失
