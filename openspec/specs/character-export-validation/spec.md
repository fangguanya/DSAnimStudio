## ADDED Requirements

### 需求:每个角色必须生成正式验收报告
正式导出链路必须为每个角色生成结构化验收报告，至少覆盖模型、动画、纹理、技能配置、参数表和 UE 导入可用性状态。

#### 场景:角色导出结束
- **当** 一个角色的正式导出流程结束
- **那么** 系统必须生成该角色的结构化验收报告并写明各子项成功或失败原因

### 需求:角色成功判定必须由子项全通过构成
只有当模型、动画、纹理、技能配置和参数表都满足正式契约时，角色才可以被记为成功。任何子项失败都必须导致角色整体失败。

#### 场景:角色存在单项失败
- **当** 角色导出中的任一子项失败
- **那么** 该角色必须被统计为失败，而不是部分成功

### 需求:UE 可用性必须进入正式验收
正式验收报告必须覆盖 UE 侧可用性，包括骨骼网格、Skeleton、PhysicsAsset、动画绑定和正式输入格式匹配情况。

#### 场景:UE 导入验证完成
- **当** 角色的 UE 导入验证流程完成
- **那么** 验收报告必须记录正式交付物是否在 UE 中形成可用资产集

### 需求:角色验收报告必须记录动画解析与装配来源
正式角色验收报告必须记录足以追溯动画求解链和模型装配来源的元数据，避免把“请求的 TAE 条目”、“实际 HKX 动画”和“正式装配结果”混为一谈。

#### 场景:导出多 ANIBND、多部件角色
- **当** 角色依赖独立的动画源、技能源或规范化多部件装配
- **那么** 验收报告必须记录相关来源和求解结果，至少包括动画来源、技能来源和正式装配标识

### 需求:角色验收报告必须提供稳定 schema 与 failure code
正式角色验收报告必须使用稳定字段名和稳定 failure code，确保回归脚本、实现代码和人工诊断可以在不解析自由文本日志的前提下比较结果。

#### 场景:同一角色重复回归
- **当** 回归脚本在不同提交上比较同一角色的正式验收结果
- **那么** 它必须能直接按固定字段和 failure code 判断是否回归，而不是依赖日志文本匹配
